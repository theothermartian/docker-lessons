Part 1

The Problem Docker Solves

Before we touch a single command, we need to understand the pain that Docker was designed to cure. The best technologies don't appear out of thin air — they emerge from frustration.

"It Works On My Machine"

Picture this: you join a new engineering team. Day one, you pull down the codebase. You spend three days just getting it to run on your laptop. Wrong Python version. Missing system library. A C extension that only compiles on Ubuntu 20.04. Your senior teammate shrugs: "works on my machine."

This is not a niche problem. It has been the most painful, productivity-destroying constant in software for decades.

Why Does This Happen?

Applications don't run in a vacuum. They depend on a fragile stack of environmental factors:

🐍

Runtime Versions

Python 3.9 vs 3.12. Node 18 vs 20. One missing minor version can break everything.

📦

System Libraries

libssl, glibc, libpq — low-level C libraries that vary wildly between OS versions.

⚙️

OS Behavior

File paths, permission models, and kernel syscalls differ between Mac, Linux, and Windows.

🌿

Environment Drift

Dev → Staging → Production: each environment gets slightly different over time. Bugs appear only in production.

💡

Docker's Answer

Stop configuring environments. Start shipping environments. Package your app and everything it needs to run into a single portable unit.

Part 2

A Brief History

Docker didn't emerge from nothing. It's built on decades of OS research into process isolation.

1979

Unix chroot

The earliest form of isolation — changing the apparent root directory for a process. The seed of an idea.

2000

FreeBSD Jails

First real container-like system — isolated processes with their own filesystem and network view.

2008

Linux cgroups + namespaces

Google contributes control groups to the Linux kernel. Combined with namespaces, this becomes the foundation all modern containers build on.

2008

LXC (Linux Containers)

The first user-space interface to cgroups and namespaces. Powerful, but complex to use.

2013

Docker 0.1 — The Revolution

Solomon Hykes demos Docker at PyCon. It's a friendly, developer-focused wrapper around LXC. The crowd erupts. Docker goes viral in the tech community.

2014–2015

Industry Adoption Explodes

Google, Amazon, and Microsoft all adopt Docker. Google open-sources Kubernetes for orchestrating containers at scale.

Today

Infrastructure Standard

Containers are the default unit of deployment across cloud platforms. Docker Hub hosts over 15 million images. 92% of IT companies use containers in production.

Part 3

The Mental Model

Analogy: The Shipping Container Revolution

Before standardized shipping containers, every ship was loaded differently. Cargo had to be repacked at every port. Loading took days. Costs were enormous.

The ISO shipping container (invented in 1956) changed everything. It doesn't matter what's inside — the box is always the same size and shape. Any crane, any ship, any port on Earth can handle it.

🚢

The Core Analogy

Docker containers are the ISO shipping containers of software. It doesn't matter what's inside — the container interface is standardized. Any machine running Docker can run your container, whether it's your laptop, a cloud VM, or your home server.

Containers vs. Virtual Machines

This is the most critical conceptual distinction for beginners. Both solve isolation, but in fundamentally different ways.

⬛ Virtual Machine

Each VM carries a full OS — GBs of overhead. Boots in minutes.

🐳 Docker Containers

Containers share the host kernel — MBs of overhead. Starts in milliseconds.

The Apartment Building Analogy

🏠 Virtual Machine = House

Fully self-contained. Its own plumbing, heating, electrical. Complete isolation. Expensive to build, wasteful of land.

🏢 Container = Apartment

Private and isolated, but shares building infrastructure — plumbing, electrical, elevators. Efficient use of resources.

⚠️

When do you still want a VM?

When you need a different OS kernel entirely (running Windows on Linux), or when maximum security isolation is required. Containers and VMs aren't mutually exclusive — in production, containers often run inside VMs.

Part 4

Core Concepts

1. Images — The Blueprint

A Docker image is a read-only snapshot containing everything an app needs: code, runtime, libraries, environment variables, and config files. Images are immutable — they never change after creation.

Think of an image like a class definition in OOP. It's a template, not a running thing.

# Pull an existing image from Docker Hub
docker pull nginx

# List images on your machine
docker images

# Output:
# REPOSITORY   TAG       IMAGE ID       SIZE
# nginx        latest    a72860cb95fd   187MB
# postgres     16        c06a31b6cfa4   432MB

2. Layers — The Efficiency Engine

Images are not monolithic blobs — they're built from stacked, immutable layers. Each instruction in a Dockerfile creates one layer. This is like git commits for your filesystem.

📝 COPY app.py /app/ ~12KB
← your code changes here
built on top of ↓
📦 RUN pip install -r requirements.txt ~45MB
← cached unless requirements change
built on top of ↓
🐍 FROM python:3.12-slim ~130MB
← shared across many images
⚡

Why Layer Order Matters

Put things that change least first. If you change your code, Docker only rebuilds from that layer up — all earlier layers are served from cache instantly. A rebuild can go from 3 minutes to 3 seconds.

3. The Dockerfile — Your Build Recipe

A Dockerfile is a plain-text script that defines how to build an image. Every line creates a layer.

# Start from an official slim Python base image
FROM python:3.12-slim

# Set the working directory inside the container
WORKDIR /app

# Copy and install dependencies first (cached layer)
COPY requirements.txt .
RUN  pip install --no-cache-dir -r requirements.txt

# Copy your actual source code (changes often)
COPY . .

# Document which port the app uses
EXPOSE 8000

# Command to run when the container starts
CMD ["python", "app.py"]
# Build the image and tag it
docker build -t myapp:v1 .

# Run a container from it, mapping port 8000
docker run -p 8000:8000 myapp:v1

4. Containers — Running Instances

A container is a running instance of an image. Image is the class; container is the object. You can run hundreds of containers from one image simultaneously.

docker run  -d -p 80:80 --name webserver nginx   # start detached
docker ps                                         # list running containers
docker logs webserver                            # view output
docker exec -it webserver bash                   # get a shell inside
docker stop webserver                            # stop gracefully
docker rm   webserver                            # remove container

5. Volumes — Persistent Data

Containers are ephemeral by default — delete the container and all its data disappears. Volumes mount a directory from the host into the container, persisting data beyond the container's life.

# Named volume (Docker manages the location)
docker run -v pgdata:/var/lib/postgresql/data postgres:16

# Bind mount (explicit host path — great for dev)
docker run -v /home/user/myapp:/app myapp:v1

# List all volumes
docker volume ls

📁 Named Volumes

Docker manages storage location. Best for databases and production data that needs to survive container updates.

🔗 Bind Mounts

Point to a specific host directory. Perfect for development — edit code on your host, see changes instantly in the container.

6. Networks — Container Communication

Containers are isolated by default. Networks let containers talk to each other while staying isolated from everything else.

# Create a custom network
docker network create myapp-net

# Run containers on the same network
docker run --network myapp-net --name db postgres:16
docker run --network myapp-net --name web myapp:v1

# "web" can now reach "db" by its container name as hostname
# e.g. DATABASE_URL=postgres://db:5432/mydb

7. Registries — Image Distribution

A registry is where images are stored and shared — like GitHub but for Docker images.

🌐 Docker Hub

The default public registry. Hosts millions of official images: postgres, nginx, redis, python, node...

🔒 Private Registries

AWS ECR, GitHub Container Registry, Google Artifact Registry — for proprietary images in enterprise environments.

🏠 Self-Hosted

Run your own registry at home with the registry:2 image. Full control, no rate limits.

Part 5

Docker Compose

Almost no real application is a single container. A typical web app needs a web server, a database, a cache, maybe a background worker. Running them all with individual docker run commands is painful and error-prone.

Docker Compose lets you define your entire stack in one YAML file and launch everything with a single command.

A Real-World Example

A Python web app with Postgres and Redis:

# docker-compose.yml
services:

  web:
    build: .                        # build from local Dockerfile
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgres://user:pass@db:5432/mydb
      - REDIS_URL=redis://cache:6379
    depends_on:
      - db
      - cache
    volumes:
      - .:/app                       # live code reloading in dev

  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=pass
      - POSTGRES_USER=user
      - POSTGRES_DB=mydb

  cache:
    image: redis:7-alpine           # alpine = tiny image

volumes:
  pgdata:                           # named volume persists data
docker compose up -d       # start entire stack in background
docker compose logs -f     # follow all logs
docker compose down        # stop and remove containers
docker compose down -v     # also remove volumes (wipes data!)
docker compose exec web bash  # shell into a running service

✅ Automatic Networking

Compose creates a shared network automatically. Services reach each other by service name as hostname.

✅ Repeatable

Same configuration every time. No forgetting a -e flag. Git-commit your compose file and share with the team.

✅ Startup Order

depends_on ensures services start in the right order — database before the web server.

🎓

The Golden Workflow

New developer joins the team: git clone the repo, run docker compose up, and the entire environment is running in seconds. No 20-page setup guide needed.

Part 9

Command Anatomy

Docker commands can look intimidating at first glance. Each block below breaks a real command apart — every argument is color-coded and explained so you can read (and build) commands with confidence.

docker run — start a container

docker run -d -p 8080:80 --name webserver -v ./html:/usr/share/nginx/html -e NGINX_HOST=localhost --rm nginx:1.27
-d
Detached mode. Runs the container in the background so your terminal is free. Without it, output streams directly to your shell and Ctrl-C kills the container.
-p 8080:80
Port mapping. Forwards host port 8080 → container port 80. Format is always host:container. Visit localhost:8080 to hit the server inside.
--name
Container name. Gives the container a human-readable name instead of a random one like sleepy_bohr. Use it in later commands: docker logs webserver.
-v src:dst
Bind mount. Maps ./html on your machine into the container's web root. Changes you make to files on the host appear inside the container instantly.
-e KEY=VAL
Environment variable. Injects a variable into the container's environment. Apps read these at startup for config — database URLs, API keys, feature flags, etc.
--rm
Auto-remove. Deletes the container the moment it stops. Keeps your system clean for one-off or test containers. Skip this for long-lived services you want to restart.
nginx:1.27
Image + tag. The image to run. nginx is the name; 1.27 is the version tag. Pinning a tag avoids surprise breakage when latest updates.

docker build — create an image from a Dockerfile

docker build -t myapp:v2 -f Dockerfile.prod --no-cache --build-arg ENV=production .
-t name:tag
Tag the image. Names the resulting image. Without a tag, Docker uses latest. Good practice: use a version number or git SHA so you can roll back.
-f FILE
Dockerfile path. Specifies which Dockerfile to use. Default is ./Dockerfile. Useful when you have separate files for dev and production builds.
--no-cache
Bypass layer cache. Forces every step to re-run from scratch. Slower, but guarantees you're not using a stale cached layer — useful in CI or when debugging a build.
--build-arg
Build-time variable. Passes a variable into the Dockerfile (referenced with ARG ENV). Different from -e — only available during the build, not at runtime.
.
Build context. The directory Docker sends to the daemon as the build context. . means the current directory. COPY instructions in the Dockerfile resolve relative to this path.

docker exec — run a command inside a running container

docker exec -it --user root --workdir /app myapp bash
-it
Interactive TTY. Two flags combined: -i keeps stdin open; -t allocates a pseudo-terminal. Together they give you a usable interactive shell. Omit both for non-interactive scripts.
--user
Run as user. Overrides the default user the container was built with. Helpful when a container runs as a restricted user but you need root access to inspect or install something.
--workdir
Starting directory. Sets the working directory for the command inside the container. Without this you land wherever the image's WORKDIR is set.
myapp
Target container. The name or ID of the running container to exec into. The container must already be running — exec doesn't start a stopped container.
bash
Command to run. The program to execute inside the container. Use bash or sh for a shell. Could also be a one-liner like env, cat /etc/hosts, or any binary in the image.

docker compose up — start a multi-container stack

docker compose -f docker-compose.prod.yml up -d --build --scale web=3 --remove-orphans
-f FILE
Compose file. Specifies an alternate compose file. Default is docker-compose.yml. Teams often keep separate files for dev and prod environments.
up
Create and start. Creates networks and volumes if they don't exist, pulls or builds any missing images, then starts all services defined in the compose file.
-d
Detached. Runs all services in the background. Without it, logs from every service stream to your terminal and Ctrl-C stops the entire stack.
--build
Force rebuild. Rebuilds images from their Dockerfiles before starting, even if a cached image already exists. Essential after you change application code or dependencies.
--scale svc=N
Horizontal scaling. Starts N replicas of the named service. Here, three web containers run simultaneously — great for load testing or simple horizontal scale-out.
--remove-orphans
Clean up removed services. Stops and removes containers for services that were in the compose file previously but have since been deleted. Keeps your stack in sync with the file.
